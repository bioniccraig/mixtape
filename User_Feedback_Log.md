| 13 | Auth | Apple Music sign-in hangs | Several users report the Apple Music connect flow hanging — stuck on "Connecting…", never completes. Not reproducible on demand. | Bug | Users | High | 🛠️ Mitigated (deployed) | 22 Jun: strongest lead is the **installed PWA** — manifest is `display: standalone`, where MusicKit's popup OAuth often can't open a window and `music.authorize()` hangs forever. Correlates with shipping the M3 install prompt, and would only bite users who added the app to their home screen (explains "can't reproduce in a browser tab"). Fix: `authorize()` now races a 45s timeout + resets state with a standalone-aware message ("open in Safari/Chrome to connect"); added a 15s timeout to `/api/musickit-token`. A hang leaves no Sentry trace (nothing throws), consistent with no error logged. Proper fix if it persists: redirect-based auth, or route Apple connect through a real browser tab when standalone. Likely a real contributor to the signup→share drop-off for the Apple cohort. |
| 12 | Sharing | Tape stays in "draft" after sharing | A user reports a tape remained in the Drafts folder even after they shared it; should auto-move to Sent. No repro detail — possibly copy/paste vs direct share. | Bug | Users | High | 🔍 Investigating (instrumented) | 22 Jun: traced every in-app share path — builder **Share**/**Community** both publish (status→published) before the share sheet, and editing a draft updates the SAME row (`dbId` seeded correctly, no orphan duplicate). Library splits purely on DB `status`; drafts expose no copy-link. So no code path loses the status flip on a normal in-app share. Most likely cause: user shared the **browser URL / phone page-share while still a draft** (nothing publishes → dead `/t/` link + stays draft), matching the copy-paste hunch. Also fixed silent clipboard failures (Sentry MIXTAPE-8) with a robust `copyToClipboard` + fallback. Added a **`share_initiated`** analytics event on every share path (builder native/community, player reshare/community, library copy-link) so we can see in data which path users take — folded into the weekly metrics report. |
| 11 | Playback | Player stalls / stops when phone locks | Two issues: (a) a tape stalls when one song ends and the next fails to play; (b) playback stops every time the phone locks (started after heavy use, not reproducible now). | Bug | Craig + Users | High | ✅ Fixed (deployed) | 22 Jun. (a) Found a real **dead-lock**: the engine only reloaded a track when its load-key changed (`yt:<id>`/`am:<title>\|<artist>`), so two ADJACENT tracks with the same id/title froze playback — key is now position-aware (`yt:<side>:<index>:<id>`). Added a **14s stall watchdog** that skips any track that never reaches "playing" (covers silent load failures that fire neither end nor error); both engines emit `onPlaying` to clear it; cleared on deliberate pause. Sentry MIXTAPE-2 (YT error 150 embed-disabled) is the live trigger; now retried/skipped without freezing. (b) Root gap: **no Media Session API** — locking the phone suspended the page with no controls. Added `navigator.mediaSession` metadata + play/pause/next/prev/stop handlers (per track). Helps Apple Music engine + gives lock-screen controls everywhere; iOS still limits background audio for the embedded YouTube player (platform constraint, can't fully fix in a PWA). |
| 7 | Social | Like button not persisting | Likes don't stick: a user likes a tape, returns later and the like is gone. Creators viewing their own shared tape see no likes at all. | Bug | Users | High | ✅ Fixed (deployed) | Root cause: tapes opened via `#tape=` hash links have no `dbId`, so likes failed the `reactions.tape_id` FK — and `toggleReaction` **swallowed the error**, so the heart filled then vanished on reload. Watch Sentry for any secondary cause (missing profile row → would need a profiles INSERT policy). |

## Shipped — 22 Jun 2026 session
Weekend user feedback → playback robustness, lock-screen controls, Apple sign-in, share instrumentation. All deployed in one push.
- **Player stall (#11a):** position-aware load key (fixes a dead-lock on adjacent same-id/title tracks) + a 14s stall watchdog; both engines emit `onPlaying`.
- **Lock-screen (#11b):** added the Media Session API (lock-screen controls + metadata). iOS YouTube background play remains a platform limit.
- **Apple sign-in hang (#13):** 45s timeout on `authorize()` + 15s on the token fetch; standalone-PWA-aware error. Root suspect: installed PWA can't open the OAuth popup.
- **Share/draft (#12):** robust `copyToClipboard` (fixes MIXTAPE-8 silent copy fail) + a new `share_initiated` event across all share paths, folded into the weekly metrics report.
- **Monitoring:** weekly-metrics task now reports a share-method breakdown; daily Sentry check now watches MIXTAPE-2 & MIXTAPE-8 for going quiet post-deploy.
- **Craig had already done:** Supabase OTP email templates (`{{ .Token }}`), the hidden-received-tapes migration, and a first read of the builder funnel.

## Decisions made
_When an item is actioned or rejected, note the outcome and date here so it doesn't get re-litigated._

- **18 Jun 2026 — Community/social features deferred to V2/V3.** The public community feed (#4) and friends/profiles (#6) are appealing but represent a deliberate shift from 1-to-1 gifting toward a social network. Decision: not now — revisit as a V2/V3 theme. Keep MixTape gifting-first through the current launch. (Lighter community touches like "tape of the week" #5 may be considered sooner.)
- **18 Jun 2026 — Like bug (#7) prioritised for immediate fix.**
- **19 Jun 2026 — Received tapes are removable via "Remove from my library" (hide, not delete).** Recipient doesn't own the tape; hide persists so it stays gone.
- **19 Jun 2026 — Footers are app-shell (in-flow), not fixed.** Permanent fix for the recurring footer-overlap issue. Don't reintroduce `position:fixed` action footers.
